# 10. Interview Preparation Questions & Answers

This chapter provides a list of technical interview questions and answers based on this repository, categorized by difficulty.

---

## 🟢 Beginner Level

### Q1: What is the purpose of `template.py` in this project?
*   **Answer**: `template.py` is a scaffolding helper script. It defines a list of all required folders and files for the project structure. When executed, it loops through the list, creates the directories using `os.makedirs()`, and creates empty files. This automates the setup of boilerplate folders and prevents manual creation errors.

### Q2: Why does the codebase have empty `__init__.py` files inside the directories?
*   **Answer**: An `__init__.py` file tells the Python interpreter that the directory should be treated as a package. This enables dot-notation imports across directories (e.g., `from src.logger import logging`). It also allows packages to run initialization code when imported.

### Q3: What is the role of `setup.py` and `requirements.txt`?
*   **Answer**:
    *   `setup.py` uses `setuptools` to package the `src/` directory as a distributable package.
    *   `requirements.txt` lists the external libraries the project depends on. The `-e .` line at the end tells pip to install the local package (`src`) in editable mode, allowing changes to code inside `src/` to be immediately available during execution.

## 🟡 Intermediate Level

### Q4: Why are different scaling methods (`StandardScaler` and `MinMaxScaler`) applied to different columns in the dataset?
*   **Answer**:
    *   `StandardScaler` is applied to `Age` and `Vintage` to standardize features by removing the mean and scaling to unit variance. This works well for features that follow a roughly normal distribution.
    *   `MinMaxScaler` is applied to `Annual_Premium` because premiums can vary widely and contain extreme values. Normalizing them to a strict `[0, 1]` range keeps values on the same scale as binary features while preserving the relative distance between outliers.

### Q5: How does the custom exception handler capture file names and line numbers?
*   **Answer**: The custom handler uses Python's standard `sys` module. When an exception occurs, calling `sys.exc_info()` returns a tuple containing information about the exception. The third element is the traceback object (`exc_tb`). The handler accesses the execution stack frame using `exc_tb.tb_frame.f_code.co_filename` to get the filename, and `exc_tb.tb_lineno` to get the line number where the exception occurred.

### Q6: Explain the difference between `config_entity` and `artifact_entity`.
*   **Answer**:
    *   `config_entity` contains configuration classes (defined as dataclasses) that specify inputs and outputs for each pipeline stage (e.g., file paths, directories, and thresholds).
    *   `artifact_entity` contains dataclasses that hold the outputs generated by each stage (e.g., split CSV paths, preprocessor file paths, and metrics). They act as a data contract between stages.

## 🔴 Advanced Level

### Q7: Explain how the `MyModel` class bundles preprocessing and prediction.
*   **Answer**: `MyModel` is a wrapper class that stores both the fitted preprocessing pipeline and the trained Random Forest classifier. When `predict()` is called, it:
    1.  Applies the saved preprocessing transformations (`preprocessing_object.transform(dataframe)`) to scale incoming features.
    2.  Passes the scaled features to the classifier (`trained_model_object.predict()`) to generate predictions.
    This ensures that preprocessing steps remain consistent between training and prediction, preventing feature mismatch errors.

### Q8: What is SMOTEENN, and why is it used instead of simple oversampling?
*   **Answer**: SMOTEENN is a hybrid resampling technique that combines SMOTE (oversampling) and ENN (undersampling).
    *   **SMOTE** creates synthetic samples for the minority class (`Response = 1`), balancing the dataset. However, this can introduce noise near decision boundaries.
    *   **ENN** cleans up this noise by analyzing the nearest neighbors of each sample. If a sample's class differs from the majority of its neighbors, it is dropped.
    Using both techniques balances the dataset while maintaining clear decision boundaries, reducing false positives.

### Q9: Why does the prediction pipeline use lazy loading for the model?
*   **Answer**: In `prediction_pipeline.py`, the model estimator is initialized to `None` in the constructor. The model is only loaded when the first prediction request is made. This lazy loading pattern reduces startup time for the web server and prevents crashes during startup if S3 is temporarily unreachable.

## 👑 Senior / Staff Level

### Q10: What architectural limitations or bugs do you see in the current codebase, and how would you resolve them?
*   **Answer**:
    1.  **Training Accuracy Evaluation Bug**: In `model_trainer.py`, the model's accuracy check compares accuracy on the *training set* against the threshold:
        `accuracy_score(train_arr[:, -1], trained_model.predict(train_arr[:, :-1])) < expected_accuracy`
        This can lead to overfitting, as a model can overfit the training data to pass the threshold check while performing poorly on unseen test data.
        *Fix*: Evaluate accuracy on the validation or test set (`test_arr`) instead.
    2.  **Deployment Port Conflict**: The GitHub Actions CD workflow runs the new container using `docker run -d -p 5000:5000` without stopping or removing the currently running container. Subsequent deployments will fail because port 5000 is already in use.
        *Fix*: Stop and remove the existing container before running the new one:
        `docker stop vehicle_app || true && docker rm vehicle_app || true`
    3.  **JSON Output in YAML File**: `data_validation.py` writes its validation report in JSON format using `json.dump()`, but the output file name is configured as `report.yaml`. This can cause errors for components expecting YAML formatted files.
        *Fix*: Use `yaml.dump()` to write the report in YAML format.

### Q11: How would you scale this pipeline to handle 100GB of vehicle data?
*   **Answer**: The current pipeline is built on pandas and scikit-learn, which process data in memory. This setup will fail with 100GB of data. To scale the pipeline:
    1.  **Distributed Processing**: Replace pandas with **PySpark** or **Dask** for data ingestion, validation, and transformation.
    2.  **Model Training**: Replace scikit-learn's Random Forest with **SparkML** or use a distributed gradient boosting library like **XGBoost on Ray**.
    3.  **Cloud Storage**: Fetch data directly from AWS S3 or a data lake instead of querying MongoDB Atlas, which can be slow for large datasets.
    4.  **Feature Store**: Implement a dedicated feature store (e.g., **Feast**) to manage feature sets and support batch training and real-time inference.
